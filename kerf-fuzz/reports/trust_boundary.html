kerf-fuzz · metamorphic + differential slicer fuzzing · report 01

The contribution isn't the bugs. It's the triage.

kerf denotes real slicer G-code back to deposited material and holds mechanically-checked metamorphic invariants — so we can transform a mesh, re-slice, and check the material transformed the same way, with no ground truth. Differential fuzzing's known weakness is false-positive triage. Pointed at PrusaSlicer 2.9.6 and Cura 5.13, this run produced 269 raw signals and adjudicated 261 of them away — including several bugs in its own harness — leaving one genuine slicer defect and one quantified fidelity boundary. What follows is what survived, and how.

01

The defect: CuraEngine isn't deterministic on doubly-curved surfaces

reproducible · ~30µm

A deterministic slicer must give byte-identical geometry for identical input. Frozen an STL to one file, md5 re-verified before every slice, single-threaded (−m1): CuraEngine produces run-to-run differences that PrusaSlicer, on the same bytes, does not. The mundane causes are experimentally excluded — it persists with one thread and with ASLR disabled (address verified constant), so it is neither threading nor address-layout; the residue is most consistent with an uninitialized-memory read. Two honesty corrections a hunt + a hard look forced:

  • Magnitude: ~30µm, not ~200µm. The coarse (200µm-grid) fingerprint I first used inflated it ~10× — a ~30µm jitter flips a whole 200µm cell. Measured on a fine grid the true run-to-run displacement is ~28µm (sphere), ~40µm (torus) — real (flat controls are exactly 0µm) but sub-cell, unlikely to change a physical print.
  • Trigger: doubly-curved surfaces, not "spheres" or "poles." A torus — no poles, no polar caps — is just as nondeterministic. Flat / single-curvature (ruled) shapes — box, cylinder, cone — are bit-exact. So the trigger is non-developable (doubly-curved) surface segments generally.
# frozen bytes, single thread, ASLR pinned — worst run-to-run divergence at a FINE (20µm) grid: box, cylinder, cone → 0 µm // bit-exact (flat / ruled) sphere → ~28 µm // doubly-curved torus (no poles) → ~40 µm // falsifies "it's the poles" PrusaSlicer, any → 0 µm // deterministic on the same bytes # mechanism excluded, not asserted: -m1 -> still diverges; ASLR off (addr pinned) -> still diverges # 15 shapes across segment counts / sub-layer radii / non-spherical curved solids = ONE bug, not 15

Minimal reproducer committed as reports/cura_nondeterminism_sphere.stl. Filable — small in magnitude, but a determinism violation is a determinism violation, and a flat-shape control isolates it cleanly. Diminishing returns on more black-box instances now; the honest next step is a sanitizer build to catch the read.

02

The boundary: thin-wall rotation, quantified

known behavior · new measurement

Rotating the input should rotate the output. Below ~1.5× the nozzle, Cura deposits a straight wall ~700 µm differently depending on orientation; PrusaSlicer stays at 0. The gap is resolution-independent (holds to a 25 µm grid), so it's a real geometric divergence — but it is not a Cura-specific defect. Both engines run Arachne (PrusaSlicer ported it from Cura in 2.5), and orientation-sensitivity at the 2→1 perimeter transition is a known Arachne characteristic. Equalizing the obvious Arachne parameters (min_bead_width, min_feature_size, wall_transition_length) did not reconcile the two — and that is itself the tell: it isn't a settings difference, it's a substrate one. CuraEngine computes on an integer-micron grid (ClipperLib), so a rotated wall quantizes before Arachne ever runs; PrusaSlicer's port doesn't share that grid. Same algorithm, different coordinate substrate — still not a Cura defect. The existence isn't novel; the quantification is.

04

What scaling to 1,680 units exposed

85 shapes was a smoke test. Scaling revealed that most of the work is hardening the harness so a verdict is trustworthy — every lesson below is now fixed or documented:

Threads

Pin the scheduler for determinism

A multithreaded slicer measured under a parallel campaign conflates slicer nondeterminism with load-induced nondeterminism. Pin one thread, or the gate is meaningless.

Freeze bytes

Hash the input before you accuse

Regenerating a mesh per slice risks blaming the slicer for your own non-bit-stable pipeline. Freeze the STL, verify its hash each run — the check that promoted Finding 1 from suspect to filable.

Topology

bbox differential ≠ disjoint solids

Comparing deposited bounding boxes is sound for one body; multi-body STLs need per-component comparison — slicers legitimately differ on placement and selection.

Exclude, don't assert

Name a mechanism only after ruling out the mundane

"Nondeterministic → ASLR" was the obvious story and it was wrong: single-thread + pinned address still diverged. Disprove threading and ASLR before you write the cause in an issue.

05

What was built to get here

  • Config-space mutation — layer-height × wall-count × seam, mapped to each slicer's native keys, swept across all 1,680 units.
  • A second real slicer + differential — CuraEngine 5.13 wired in (self-healing its unresolved-default settings), enabling cross-slicer comparison metamorphic testing alone can't provide.
  • An EMI relation — a semantics-preserving mesh mutant must slice to identical material.
  • A parallel campaign runner — geometry × config × slicers, streamed to a resumable checkpoint.
  • Corpus where bugs live — thin walls straddling the nozzle, hole/ligament tolerance, tall seam-stress, multi-island, bridging, near-degenerate facets.

Depth over volume: two airtight results beat a bigger corpus. This is a smoke test toward a real campaign (10⁴–10⁵ + deeper config mutation), not a "clean" verdict on either slicer. Every number is reproducible from the committed harness; the self-validation oracle held at 72/72, so a surviving signal is about a slicer, not the tool.
