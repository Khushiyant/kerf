Design Document · Verified Oracle Fuzzing

A slicer fuzzer built on kerf, the semantic oracle the mesh→G‑code compiler never had.

Mainstream slicers — PrusaSlicer, CuraEngine, OrcaSlicer — are compilers from solid geometry to machine code, yet they have had neither a Csmith (a generator of well-defined inputs) nor an EMI-style equivalence oracle. kerf supplies both halves: a per-layer deposited-material denotation with machine-checked metamorphic invariants, driving an unbounded metamorphic + differential fuzzing loop that finds real bugs.

The gap

Slicers have no reference semantics, so bugs hide as "plausible G-code." GlitchFinder (OOPSLA 2025) took the first step — differential-only, over a fixed 58-model corpus.

The unlock

kerf denotes G-code to per-layer occupancy and proves that occupancy is translation- and rotation-equivariant. That makes an EMI-style metamorphic oracle possible.

The discipline

Every verdict is modulo resolution r, graded not binary where rounding bites, and gated on a determinism check. Skepticism about false positives is the whole design.

A

Architecture

A single feedback loop: generate or mutate a mesh, apply an isometry, drive it through pluggable slicer adapters, denote each output with kerf, check the invariant, then shrink and report any surviving violation.

01 · SOURCE
Mesh gen / mutate
CSG primitives via manifold3d; valid-by-construction, watertight. Boundary-targeted parameter sweeps.
02 · TRANSFORM
Isometry
Whole-cell translate, 90°/arbitrary-Z rotate, mirror. Applied at mesh and at parsed-program level.
03 · COMPILE
Slicer adapters
Pluggable: Prusa / Cura / Orca. One control profile, one binary per pair, pinned determinism flags.
04 · ORACLE
kerf parse + denote
G-code → lo::Program → Occupancy / Deposit / Volume / Flow, matched by layer Z.
05 · CHECK
Invariant
Exact fingerprint, graded EDT distance, or IoU vs the relation's tolerance ε.
06 · REDUCE
Shrinker → report
Delta-debug faces to a minimal witness; attribute which denotation moved.

Two loops share one oracle

The metamorphic loop compares a slicer against itself under a geometric symmetry — this is the sound core, because it eliminates the entire class of "legitimate slicer choice" false positives (seam, infill pattern, wall order) that a naive differential harness drowns in. The differential loop compares two slicers on the same mesh, but only on what geometry forces — the outer shell envelope — as a graded containment check.

Both consume the same parse() → denote_lo path. The oracle sees the set of raster cells that extruding toolpaths' swept capsules (polyline ⊕ disk of width_um) touch, unioned per layer and matched by layer Z, not by line order. That order-insensitivity is what lets it survive path reordering, arc-vs-segment representation, and multithread scheduling — differences that are real in the bytes but semantically empty.

Design invariant No stage ever fires a verdict on a difference that lives only in re-inferred metadata. Toolpath role and width are re-inferred by the parser from ;TYPE: / ;WIDTH: comments and are untrusted. Before any relation is admitted, the harness checks kerf Diagnostics: if fallback_role_moves, estimated_width_moves, or unknown_roles touch the cells in question, the "bug" is inadmissible.
B

The sound invariant set

Every relation is filed under a soundness class and carries its own false-positive controls. A "DIFFER" verdict is only as trustworthy as the class it comes from — and only as sharp as the resolution r it was measured at.

SOUND-EXACT

Transform is a true symmetry of the physical part; kerf backs it with an exact integer relation. A violation beyond tolerance is necessarily a slicer defect.

SOUND-GRADED

Sound in principle, but the transform induces sub-cell rounding. Obligation is p95_um ≤ ε, not exact equality. Report near-misses; flag only past ε.

DIRECTIONAL

Transform legitimately changes the denotation in a predicted way (volume scales). Sound only as a monotonicity / conservation law — never as equality.

GATE

Not a bug detector. The precondition (determinism) that every other verdict depends on. If it fails, downstream results are uninterpretable.

The one load-bearing caveat Every verdict is modulo resolution r. "SAME" means "same deposited material up to r"; "DIFFER" means separable at r. Cell inclusion is conservative coverage (a capsule reaching within a cell's circumradius r/√2 includes it), so a feature is never missed when r ≤ its width — the oracle over-includes, never under-includes. Choose resolution_um ≤ the smallest feature you are willing to call a bug.

Why the core is metamorphic, not differential

M1, M2a, and M4 rest on relations kerf has proved — translation-invariance (P2) and 90° rotation-invariance are machine-checked in Lean 4, and reversal-invariance (P1) is why occupancy survives a mirror's direction flip. The relation's soundness therefore reduces to the physical symmetry argument, not to trusting kerf's rasterizer code. That is a stronger footing than any cross-slicer comparison can offer.

Traps — unsound as equality Never compare full-program occupancy across a scale (M5a), a layer-height change (M5b), or two different slicers. Scaling changes volume by design; a layer-height change makes the Z's disjoint (kerf correctly reports them as non-overlapping); and two correct slicers differ legitimately on infill, seam, support, and arc representation. These are conservation laws, not equalities.

Per-relation false-positive controls

  • M1-e2e: shift by a whole number of cells only (Δ = c·r µm) — a non-cell-multiple shift aliases the grid and is not a clean relation. Keep the translated part fully on-bed; an off-bed clamp is legitimate.
  • M2a / M2b: infill angle is world-referenced in every slicer, so a fixed angle after rotation yields a rotated pattern that is not the rotation of the original infill. Restrict to the deterministic solid region (perimeters + skin), or set infill to co-rotate (Orca align_infill_direction_to_model=true), or raise density to 100%.
  • M2b budget: ε must cover rasterization at r, the ≤1µm per-vertex rounding rotate_z applies, and the slicer's own µm quantization. Defensible: ε = r + 2·√2·1µm ≈ r + 3µm, checked on p95_um.
  • M4: reflection flips chirality — seam side, wall travel direction, infill phase. Occupancy is direction-agnostic so it survives, but mask the seam. Honesty note: kerf ships no native reflect primitive — the axis-negation map is a ~10-line harness helper; only the comparison is a kerf primitive.
C

Per-slicer adapter templates

Verified command templates and determinism flags. Low-confidence items are flagged inline — config keys must be re-confirmed against the exact build's --help-fff / settings.def.json at harness-setup time; slicers drop and rename keys between releases.

PrusaSlicer 2.9.6 stable · Jun 2026
confidence: high

An action is required — --export-gcode (unlike old Slic3r, a bare STL does nothing). On Windows use prusa-slicer-console.exe. Compare only G-code from the same binary on the same machine; cross-build float drift will false-fail byte comparisons.

prusa-slicer --export-gcode --load PROFILE.ini --output OUT.gcode \
  --dont-arrange --center 100,100 --threads 1 \
  --arc-fitting disabled --fill-pattern concentric \
  --seam-position random --resolution 0 --gcode-resolution 0.0125 \
  --spiral-vase 0 INPUT.stl

Determinism: --threads 1 (default is auto = hardware concurrency — the #1 cross-host non-reproducibility source). --arc-fitting disabled keeps all extrusion as straight G1. --dont-arrange stops the packer jostling the model. For isometry pairs, prefer --fill-pattern concentric (co-rotates with the part) and --seam-position random (fixed seed 1653710332u, fractional arc-length — rotates with geometry) over the world-anchored defaults stars / aligned.

Corrected — flagged low-confidence --loglevel: master source and the man page disagree by inversion. Source semantics say 2 = warning (not "info"); use 3 if you actually want info. Does not affect G-code bytes — verify on your build with --help. Also: the widely-cited PrusaSlicer issue #7453 ("stray G3") is miscited — that issue is unrelated; keep arc-fitting off regardless.
CuraEngine Cura 5.13 latest · engine built from main
confidence: high

No official standalone binary — build via Conan or extract from a matching Cura install; keep engine and definitions version-matched. You must -j fdmprinter.def.json before any -s that references a key, or 5.x aborts with "Trying to retrieve setting with no value given" (the #1 empty-gcode cause). Argument order matters: -s applies to the last scope selected by -e/-l/-g.

CuraEngine slice -v -m1 -j <dir>/fdmprinter.def.json -d <dir> \
  -s adaptive_layer_height_enabled=false -s coasting_enable=false \
  -s z_seam_type=back -s z_seam_relative=true \
  -s magic_fuzzy_skin_enabled=false  # NOT fuzzy_skin_enabled \
  -s infill_pattern=grid -s infill_offset_x=0 -s infill_offset_y=0 \
  -s connect_infill_polygons=false  # real default is TRUE \
  -s center_object=false -s mesh_position_x=0 \
  -e0 -l model.stl -o out.gcode

Emits only G0/G1 — there is no native arc setting (magic_arcs does not exist; G2/G3 arise only via the external ArcWelder plugin, so leave it out). Works in integer microns internally, so expect sub-cell rounding on ~45° rotations — exactly what M2b's graded distance absorbs. Feed transforms through mesh_rotation_matrix or a pre-transformed STL and keep center_object=false.

Corrections that would otherwise silently break the harness The fuzzy-skin key is magic_fuzzy_skin_enabled — the bare fuzzy_skin_enabled is an unknown key and has no effect. connect_infill_polygons defaults to true, so set it false explicitly. Also consider the single-file resolved-profile path -r resolved.json instead of the -j+-s merge.
OrcaSlicer / Bambu Studio shared CLI · SoftFever main
confidence: high, w/ caveats

There is no --export-gcode flag (commented out in source). Plain G-code appears only when --outputdir is set (writes plate_N.gcode); otherwise it is buried in --export-3mf as Metadata/plate_1.gcode. No --threads flag — pin TBB with taskset -c 0 on Linux. Presets use inherits chains; a raw child JSON is incomplete — flatten inheritance yourself.

taskset -c 0 "$ORCA" --slice 0 \
  --load-settings "$MACHINE_JSON;$PROCESS_JSON" \
  --load-filaments "$FILA_JSON" --outputdir "$OUTDIR" --debug 2 input.stl
# → $OUTDIR/plate_1.gcode  (+ result.json on Linux only)

Determinism keys (in process JSON): enable_arc_fitting=false, seam_position=aligned (never random), fuzzy_skin=none, and align_infill_direction_to_model=true so infill co-rotates (default is false = world-anchored). Use --arrange 0 and omit --orient; use --rotate θ for an exact Z rotation.

Three corrections vs. the raw report (1) --load-settings order is not load-bearing — files dispatch by their internal type field, not CLI position; supply one machine-typed and one process-typed JSON. (2) --no-check disables validity checks (incl. the G-code path-conflict detector — a bug class you may want) — do not pass it reflexively. (3) result.json is written on Linux only; run the harness on Linux/WSL. Avoid multi-material painted 3MFs (segfault, issue #12426).
D

Validating the harness with zero external slicers

Before pointing the fuzzer at any real slicer, prove the harness itself is correct — using kerf's own lower → G-code emitter as a reference "slicer" whose metamorphic relations must pass exactly.

kerf can emit G-code from its own lo::Program (guarded per-program by verify_roundtrip). Treat that emitter as a fourth, in-process slicer adapter. Because it is deterministic and its geometry is exactly what kerf denotes, it is the ideal oracle-under-test: every SOUND-EXACT metamorphic relation must pass bit-exactly on it, and every graded relation must pass at max_um = 0 for whole-cell / 90° transforms. Any failure is a harness bug — in the transform composition, the region mask, the diff wiring, or the fingerprint — not a slicer bug.

The self-test matrix

  • M0 must be trivially exact: re-emitting the same program twice yields identical material_fingerprint. If not, the emitter or hash is nondeterministic — stop.
  • M1-int is a proven tautology: translation_invariant holds by Lean P2, so a failure means the harness applied translate or read Occupancy incorrectly.
  • M2a must be exact index-permutation: voxelize(reference(rot90(mesh))) == rot_z90(voxelize(reference(mesh))). This exercises the whole voxelize + rotate + compare path with a known-good answer.
  • M4 round-trip: reflect twice = identity; the harness reflection helper must compose to diff.identical against the original.
  • Round-trip metric floor: rotate_z(rotate_z(p, θ), −θ) must land at mean_um ≤ r — the same bound kerf's own property tests assert.
Why this matters This gives a full CI gate with no external dependencies. A green self-test suite means the shrinker, the region masker, the isometry composition, and the tolerance math are all sound; only then is a "DIFFER" against PrusaSlicer attributable to PrusaSlicer. It also directly validates the differential machinery: two reference re-emissions of the same program must yield IoU = 1.0 and max_um = 0 on the shell mask.

Second use: mutation testing of the harness. Deliberately perturb a reference program (drop a path, shift one cell, widen a bead) and confirm each relation fires with the expected verdict and severity — occupancy for a dropped path, volume_stats for a widened bead, deposit_stats for a duplicated path. This calibrates the "which denotation moved" attribution before it is trusted on real bugs.

Bug-class targets & mesh generation

What to hunt, why kerf can see it, and how to generate the inputs that provoke it. Generate at the CSG level and export valid STL — byte-fuzzing STL is rejected ~65% of the time as invalid.

Mesh generation strategy

Build the "Csmith for meshes": compose boxes, cylinders, spheres and extrusions via boolean union / difference / intersection with randomized parameters, using manifold3d as the backend so every CSG output is provably manifold by construction. Gate validity with trimesh (is_watertight, is_winding_consistent, degeneracy checks).

  • Valid-by-construction track: watertight, manifold, features > nozzle width — so a divergence is unambiguously a bug (the Csmith "avoid undefined behavior" analogue).
  • Boundary-targeted sweeps: sweep wall thickness across nozzle width, overhang angle across the bridging threshold — bisects the exact point a slicer drops a feature, yielding crisp reports.
  • Semantics-preserving mutations (the EMI analogue): rigid translate, 90°/Z rotate, remesh a face, insert redundant vertices, merge coplanar faces — occupancy must not move past tolerance.
  • Malformed track (robustness only): vertex jitter into self-intersection, hole creation, normal flips, sliver-triangle injection.
Region masking makes E + B work together Confine M2a/M2b/M4 and the differential shell check to the deterministic region: filter lo::Program toolpaths to kind ∈ {Perimeter, Skin}, drop Infill/Support, then diff. This removes infill-pattern and support freedom as false-positive sources — but only mask on roles the parser knew (from a recognized ;TYPE: token, not a fallback_role_move). If roles were guessed, fall back to a whole-program graded comparison with wider ε, or skip the config.
F

Phased build order

What to build first with no external slicers, and what waits on the user's Linux slicer binaries. Each phase gates the next.

PHASE 0
no slicers needed

Harness skeleton + self-validation (§D)

Wire the reference "slicer" (kerf's lower → G-code) as the first adapter. Implement the transform helpers (translate, rot90, rotate_z, the reflection helper), the region masker, and the verdict pipeline against it.

  • All SOUND-EXACT relations pass bit-exactly on the reference; graded relations hit their mean_um ≤ r floor.
  • Mutation-test the harness: perturb a reference program, confirm the expected relation fires with the expected severity.
  • Exit gate: green CI with zero external dependencies.
PHASE 1
no slicers needed

Mesh generator + shrinker (§E)

CSG generator on manifold3d, validity gate on trimesh, boundary-targeted sweeps, and the delta-debug face shrinker. Validate the shrinker by feeding known-divergent reference programs and confirming it minimizes to the expected witness.

  • Corpus of valid-by-construction meshes + a labeled malformed set.
  • Shrinker produces < 20-triangle witnesses on seeded failures.
PHASE 2
needs 1 Linux slicer

First real adapter + metamorphic loop (§B, §C)

Bring up PrusaSlicer on Linux (cleanest CLI, best-verified flags). Run M0 first on every config; only if determinism holds, run M1 / M2a / M2b / M4 self-comparison. This is where the first real bugs appear.

  • Establish the fixed control profile; confirm M0 passes under it.
  • Two-resolution confirmation + Diagnostics admissibility gate live in the loop.
PHASE 3
needs 2–3 Linux slicers

Differential loop + remaining adapters (§C, §A)

Add CuraEngine and OrcaSlicer (Linux/WSL — Orca's result.json is Linux-only). Enable the differential shell-containment check (D0) and the volume band (D2). Every adapter reuses the Phase-2 pipeline unchanged.

  • Match perimeter count and infill density across slicers before D0/D2.
  • Differential is graded and region-masked only — never full-program equality.
PHASE 4
needs slicers + repair tools

Campaign, triage, report

Continuous fuzzing with boundary sweeps; auto-triage by which denotation moved; the repair-tool differential track (MeshLab / Meshmixer as systems-under-test). Package minimized witnesses with per-invariant provenance for upstream bug reports.

G

Contribution & threats to validity

Honest framing. The strongest prior art is close — that is a reason to sharpen the claim, not abandon it.

The contribution, stated precisely

Slicers are compilers from solid geometry to machine code, yet unlike C compilers they have had neither a Csmith (a generator of well-defined inputs) nor an EMI-style semantic oracle. GlitchFinder (OOPSLA 2025) took the first step — denoting G-code to occupancy for differential slicer comparison over a fixed 58-model corpus — but it is differential-only, corpus-bound, uses an approximate point cloud, and has no proven equivalences.

kerf supplies the two missing halves: a semantic denotation with machine-checked metamorphic invariants (translation- and rotation-equivariance, with a graded nearest-cell distance and IoU that absorb sub-cell rounding), and a procedural mesh generator that turns those invariants into an unbounded metamorphic + differential fuzzing loop. This is the first setup where both the Csmith (generator) and the EMI (proven equivalence oracle) exist for the mesh→G-code compiler.

Delta vs. GlitchFinder Three things GlitchFinder lacks and kerf has: (1) proven metamorphic equivariances of the denotation itself, not just invariant-checking as a use case; (2) a generator, so this is a fuzzer rather than a study over a fixed corpus — the analogue of testing GCC only on SPEC and never Csmith; (3) graded, tolerance-aware metrics rather than an approximate cuboid point cloud. That GlitchFinder independently arrived at the denotation-as-oracle idea is the strongest possible validation that the approach is real and publishable.

Threats to validity — where a "bug" might not be one

  • Resolution-bounded oracle. Every verdict is modulo r; a defect smaller than r is invisible and a "SAME" is only "same up to r." This is definitional, not a flaw — but it bounds every claim.
  • Geometry, not process. No temperature, cooling, pressure-advance, retraction, or timing. Thermal and retraction bugs are out of scope; print_time is a model, not the oracle.
  • Re-inferred metadata. Role and width are untrusted parser inferences; the Diagnostics gate must exclude any verdict living only in guessed metadata, or false positives leak in.
  • kerf's own soundness is layered, not a single mechanized geometry proof. P1–P4 are Lean-proved unbounded, the rasterizer's symmetry is Kani-proved, and the rest is property-tested. Strong enough that M1/M2a rest on proved behavior — but the emitter used in §D is guarded, not part of the verified definition.
  • Determinism assumption. Every metamorphic verdict presumes M0 passed under the control profile. If a slicer is irreducibly nondeterministic under those flags, that config yields no interpretable verdict (and the nondeterminism is itself a finding).
  • Differential is inherently weaker. Its only clean signal is outer-shell graded containment and a volume band; full cross-slicer occupancy equality is unsound because of legitimate infill / seam / support / arc freedom.
  • Version drift. All CLI flags are pinned to specific builds (Prusa 2.9.6, Cura 5.13, Orca main); config keys must be re-confirmed per build. Pin exact tags in the harness.
Why the core claim survives every threat above The threats degrade the differential loop and bound the resolution, but the sound core — metamorphic self-comparison under a proved symmetry, gated on determinism, graded where rounding bites, minimized before reporting — produces violations that are, by construction, attributable to the slicer and reproducible. That is a genuinely new capability for this pipeline.
